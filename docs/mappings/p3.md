# P³ layer mapping

Status vocabulary: mature, partial, missing, not applicable.

| Layer | Status | Evidence and gap |
| --- | --- | --- |
| Identity | mature | P³ already has a clear personal pipeline role: capture, curate, and turn raw personal signals into reusable briefs, project ideas, and decisions. The scaffold should preserve that product/research operating identity instead of turning P³ into a generic notes app. |
| Context | mature | Source snapshots, project-ideas issues, repo-local plans, and closeout comments already act as context packets. The next mapping step is to keep each generated execution repo linked back to its source issue and plan. |
| Skills | partial | Reusable skills exist as repeated protocols: repo bootstrap, issue-by-issue execution, validation briefs, and closeout retrospectives. They are still partially implicit and should be promoted into repo-local docs or Hermes skills when repeated. |
| Memory | partial | Durable lessons exist in retrospectives and memories, but retrieval boundaries are uneven. Future work should separate stable project facts from transient execution state. |
| Connections | mature | GitHub repos, issue backlogs, local files, CLI tools, and generated artifacts are explicit connections. Authentication must stay outside committed artifacts. |
| Verification | partial | Verification is not trust-by-narrative: concrete evidence includes the `mlx-grammar` fixture eval runner, optional-backend smoke-test gate, `signal-newsletter` quality gates, repo-local pytest/Ruff commands, independent review before commit, and final closeout checks. Future P³ work should link each automation to an executable check or evaluation artifact before it is treated as mature. |
| Automations | missing | Scheduling and event triggers are not the default maturity path yet. Add automations only after the corresponding Verification row has deterministic checks and rollback notes. |

## Sequencing guidance

1. Put new source ideas and plans in Context first.
2. Promote repeated execution protocols into Skills only after they have been used more than once.
3. Require a Verification issue before adding any Automation issue.
4. Treat Memory as durable only when the fact will influence future decisions.
